# 视频推流与摄像头管理方案

本文档记录下位机视频推流、上位机五窗口显示切换，以及后续两路 OpenCV 后处理画面的接入方案。

当前结论：

- 视频像素流继续走 RTSP，不走 TCP/JSON。
- ROS/TCP JSON 只负责视频源列表、状态、启动停止请求和错误信息。
- 不需要后处理的摄像头优先直接从硬件设备推 RTSP。
- 需要 OpenCV 后处理的两路图像可以走 ROS topic 作为算法输入，但处理后画面应尽量由 OpenCV 节点直接推 RTSP，避免再做一层 ROS 图像中继。

## 1. 总体架构

```text
上位机 Qt
  |-- TCP/JSON: camera_list_request / camera_info / camera_stream_request
  `-- RTSP 拉流: rtsp://下位机:8554/<stream_path>

下位机 ROS1
  |-- host_bridge_node
  |     |-- 响应 camera_list_request
  |     |-- 转发 camera_info
  |     `-- 转发或处理 camera_stream_request
  |
  |-- video_manager_node
  |     |-- 读取摄像头配置
  |     |-- 启停无处理直推管线
  |     |-- 监控推流进程健康
  |     |-- 生成 camera_info / camera_list
  |     `-- 管理 RTSP server / 推流进程
  |
  |-- 直接视频源
  |     `-- /dev/videoX / CSI / 网络摄像头 -> RTSP
  |
  `-- 后处理视频源
        `-- ROS compressed topic -> OpenCV C++ 节点 -> RTSP
```

## 2. Camera Source 和 Display Slot 分离

上位机有五个显示窗口，但显示窗口不应直接等同于物理摄像头。

- `CameraSource`：真实视频源，例如 `front_raw`、`rear_raw`、`arm_processed`。
- `DisplaySlot`：上位机五个显示槽位，例如 `slot_id = 0..4`。

窗口切换时，本质是：

```json
{
  "slot_id": 2,
  "camera_id": "front_raw"
}
```

短期可以先在上位机本地完成 slot 到 camera 的映射；下位机只提供所有可用 `CameraSource` 的列表和 RTSP URL。

## 3. 摄像头信息模型

`camera_info` 和 `camera_list_response.cameras[]` 建议统一使用以下字段：

```json
{
  "camera_id": "front_raw",
  "slot_hint": 0,
  "name": "Front Raw",
  "kind": "direct",
  "online": true,
  "rtsp_url": "rtsp://192.168.1.50:8554/front_raw",
  "codec": "h264",
  "width": 1280,
  "height": 720,
  "fps": 30,
  "bitrate_kbps": 2500,
  "profile": "low_latency",
  "last_error": ""
}
```

说明：

- `camera_id` 使用稳定字符串，不再把它强绑定为 UI 槽位编号。
- `slot_hint` 只是默认显示建议；上位机可以忽略并允许用户切换。
- `kind` 可取 `direct`、`processed`、`network`、`debug`。
- 兼容期内，当前上位机仍只接受 `camera_id` 为 `0..4` 的整数；真正做 slot/source 解耦时需要升级 schema。

## 4. 第一阶段：无处理直推 RTSP

第一阶段只做不需要 OpenCV 后处理的摄像头。目标是先把“摄像头硬件 -> RTSP -> 上位机”跑通。

### 4.1 职责边界

`video_manager_node` 负责：

- 读取摄像头配置。
- 为每路无处理摄像头启动一个推流管线。
- 监控推流进程是否退出。
- 定期或事件触发生成 `camera_info`。
- 响应 start / stop / restart 请求。

`host_bridge_node` 负责：

- TCP/JSON 协议收发。
- 响应上位机的 `camera_list_request`。
- 将 `video_manager_node` 提供的 camera list 转成 `camera_list_response`。
- 将摄像头上线/离线变化转成主动 `camera_info`。

RTSP server 负责：

- 对外提供固定 RTSP URL。
- 接收本机推流进程发布的流。
- 供上位机五窗口拉流。

### 4.2 推荐组件

优先推荐：

- RTSP server：MediaMTX。
- 推流管线：GStreamer 或 FFmpeg。
- 管理节点：Python 或 C++ ROS1 node 均可。第一版用 Python 更快，后续稳定后可改 C++。

不要做的事情：

- 不要把所有无处理摄像头先发布成 ROS `sensor_msgs/Image` 再转 RTSP。
- 不要用 Python `cv_bridge` 搬运五路原始画面。
- 不要让 TCP/JSON 承载视频帧。

### 4.3 配置示例

建议新增下位机配置文件，例如 `config/video_sources.yaml`：

```yaml
rtsp:
  host: 192.168.1.50
  publish_host: 127.0.0.1
  port: 8554
  server: mediamtx
  transport: tcp

direct_sources:
  - camera_id: 0
    source_id: front_raw
    slot_hint: 0
    name: Front Raw
    runner: ffmpeg
    device: /dev/video0
    input_format: mjpeg
    width: 1280
    height: 720
    fps: 30
    codec: h264
    bitrate_kbps: 2500
    crop_aspect: "4:3"
    rtsp_path: front_raw
    enabled: true

  - camera_id: 1
    source_id: rear_raw
    slot_hint: 1
    name: Rear Raw
    runner: ffmpeg
    device: /dev/video1
    input_format: mjpeg
    width: 1280
    height: 720
    fps: 30
    codec: h264
    bitrate_kbps: 2500
    rtsp_path: rear_raw
    enabled: true
```

### 4.4 推流管线策略

如果摄像头本身输出 H264/H265，优先尝试直接封装或 copy，避免重新编码。

如果摄像头输出 MJPEG：

- 带宽宽裕、调试阶段可以先用 MJPEG RTSP。
- 需要节省带宽时，转 H264/H265。
- 转码优先使用硬件编码；没有硬件编码时再用 `x264enc` / `libx264` 的低延迟参数。

示例命令只作为方向，最终应按下位机硬件实测调整：

```bash
# FFmpeg: UVC MJPEG -> H264 low latency -> RTSP server
ffmpeg \
  -f v4l2 -input_format mjpeg -video_size 1280x720 -framerate 30 -i /dev/video0 \
  -an -c:v libx264 -pix_fmt yuv420p -profile:v baseline \
  -preset ultrafast -tune zerolatency -g 30 -b:v 2500k \
  -rtsp_transport tcp -f rtsp rtsp://127.0.0.1:8554/front_raw
```

```bash
# GStreamer: UVC MJPEG -> H264 low latency -> RTSP server
gst-launch-1.0 \
  v4l2src device=/dev/video0 ! \
  image/jpeg,width=1280,height=720,framerate=30/1 ! jpegdec ! videoconvert ! \
  x264enc tune=zerolatency speed-preset=ultrafast bitrate=2500 key-int-max=30 ! \
  rtspclientsink location=rtsp://127.0.0.1:8554/front_raw
```

### 4.5 当前 bridge 启动方式

第一版已支持 FFmpeg direct source runner。示例配置位于：

```bash
ros1_bridge/video_sources.example.yaml
```

本机 dry-run 验证，不启动 FFmpeg，只验证 camera list、命令生成和协议：

```bash
python3 ros1_bridge/host_bridge_node.py \
  --dry-run \
  --host 127.0.0.1 \
  --port 9090 \
  --video-config ros1_bridge/video_sources.example.yaml \
  --video-dry-run \
  --video-autostart \
  --debug-ui
```

下位机真实启动 FFmpeg 推流：

```bash
python3 ros1_bridge/host_bridge_node.py \
  --ros \
  --host 0.0.0.0 \
  --port 9090 \
  --video-config /path/to/video_sources.yaml \
  --video-autostart \
  --debug-ui
```

注意：

- 非 dry-run 模式要求下位机已安装 FFmpeg。
- 如果使用 MediaMTX 作为 RTSP server，需要先启动 MediaMTX，并允许本机向 `rtsp://127.0.0.1:8554/<path>` publish。
- 默认 `transport: tcp`，即 FFmpeg 发布端通过 RTSP/TCP 传 RTP；这样只需要暴露 8554/TCP，Docker、本机回环和跨机器部署都更容易跑通。如果明确映射并放通 RTP/UDP 端口，可以把 `transport` 改为 `udp`。
- 当前上位机仍要求 `camera_id` 是 `0..4` 整数，所以第一版配置保持整数 ID；字符串 `source_id` 只作为未来 source/slot 解耦的稳定名字。
- `--video-autostart` 只启动配置中 `enabled: true` 的源。
- `crop_aspect: "4:3"` 会在下位机 FFmpeg 推流前做中心裁切。例如 `1280x720` 会裁成 `960x720`，再编码推送 RTSP；桥接节点也会把 `camera_info.width/height` 回传为裁切后的尺寸，避免上位机按原始 16:9 尺寸解码拉伸。需要更通用处理时可配置 `video_filter`，其内容会直接作为 FFmpeg `-vf` 的一段 filter；若自定义滤镜改变分辨率，需要同步调整配置或后续扩展显式输出尺寸。

## 5. 第二阶段：OpenCV 后处理画面

你当前有两路需要 C++ OpenCV 后处理的图像。这两路可以允许 ROS topic 出现在算法输入链路里。

推荐链路：

```text
摄像头节点或压缩图像源
  -> /xxx/image/compressed
  -> OpenCV C++ 后处理节点
  -> GStreamer appsrc / FFmpeg pipe
  -> RTSP
  -> 上位机
```

原则：

- 如果必须走 topic，优先 `sensor_msgs/CompressedImage` 或 `image_transport/compressed`。
- OpenCV 节点内部必然会解码成 `cv::Mat`，处理完成后直接送 RTSP 管线。
- 处理后的结果不要再发给另一个“视频中继 ROS 节点”去搬运，除非确实需要其他 ROS 节点复用这个后处理结果。
- 两路后处理可以接受一到两次 topic；五路原始视频不应这样做。

## 6. 控制协议扩展

已有：

- `camera_list_request`
- `camera_list_response`
- `camera_info`

建议新增关键命令：

```json
{
  "type": "camera_stream_request",
  "protocol_version": 1,
  "seq": 120,
  "timestamp_ms": 1710000000000,
  "params": {
    "camera_id": "front_raw",
    "action": "restart",
    "profile": "low_latency"
  }
}
```

`action` 可取：

- `start`
- `stop`
- `restart`
- `set_profile`

该命令应返回 ACK。成功后下位机主动推一次新的 `camera_info`。

上位机本地 slot 切换不一定需要下发命令；只有当某路流尚未启动、需要切 profile、或需要重启推流时，才发 `camera_stream_request`。

## 7. 第一阶段落地步骤

建议先做无处理直推，按以下 commit 拆：

1. `docs(video): document RTSP source manager plan`
   - 落本文档。
   - 在通信计划中标记视频链路第一阶段。

2. `feat(video): add direct RTSP source config`
   - 新增 `video_sources.yaml`。
   - 定义 `direct_sources`、RTSP host/port、默认 codec/profile。

3. `feat(video): add video manager direct source runner`
   - 新增 `video_manager_node.py` 或 C++ node。
   - 能启动/停止单路 FFmpeg/GStreamer 进程。
   - 能读取配置并生成 camera list。

4. `feat(bridge): expose video manager camera list`
   - host bridge 从 video manager 获取实时 camera list。
   - `camera_list_response` 不再只依赖静态 `--camera` 参数。
   - 摄像头状态变化时主动广播 `camera_info`。

5. `feat(video): support camera stream requests`
   - 增加 `camera_stream_request`。
   - 支持 start / stop / restart。
   - 返回 ACK，并刷新 `camera_info`。

6. `test(video): add direct RTSP smoke test`
   - 用测试源或本机虚拟源启动 RTSP。
   - 验证 camera list、URL 可访问、进程退出能上报离线。

## 8. 验收标准

第一阶段完成时，应满足：

- 下位机启动后，无处理摄像头能自动推 RTSP。
- 上位机连接后能收到真实 `camera_list_response`。
- 五个窗口能按 `camera_info.rtsp_url` 拉到画面。
- 任意一路推流进程退出后，下位机能上报离线或错误。
- 对某一路发送 `camera_stream_request restart` 后，能 ACK 并恢复推流。
- 不需要后处理的摄像头没有走 ROS image topic。

## 9. 上位机播放后端

当前上位机播放端使用 FFmpeg 子进程低缓冲拉流，再把 raw RGB 帧交给 Qt Widget 渲染：

```text
RTSP URL -> ffmpeg -fflags nobuffer -flags low_delay -> rawvideo rgb24 -> Qt Widget
```

部署要求：

- 构建不需要 FFmpeg 开发库，不链接 `libav*`。
- 运行时需要 `ffmpeg` 可执行文件。
- 查找顺序：`HOSTCOMPUTER_FFMPEG_PATH` 环境变量、上位机可执行文件同目录、系统 `PATH`。
- 为了 Windows/Linux 开箱即用，正式发布包应把 `ffmpeg.exe` 或 `ffmpeg` 放在上位机可执行文件同目录。详细部署检查见 [上位机部署说明.md](上位机部署说明.md)。
- 开发机如果已经能在终端执行 `ffmpeg -version`，上位机也可以直接使用系统 `PATH` 里的 FFmpeg。

## 10. 主要风险

- USB 摄像头可能不能被多个进程同时打开；直接推流和 ROS camera node 不能抢同一个 `/dev/videoX`。
- MJPEG 直推调试简单，但多路带宽可能偏高。
- H264/H265 低延迟效果依赖硬件编码能力和管线参数。
- 上位机 FFmpeg 播放后端依赖运行时 `ffmpeg`，多路同时解码时需要观察 CPU 占用。
- 多路同时拉流时，需要观察下位机 CPU、编码器占用、网络带宽和端到端延迟。
