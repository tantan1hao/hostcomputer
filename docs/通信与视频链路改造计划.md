# 通信与视频链路改造计划

本文档记录上位机与 ROS1 下位机通信、控制命令可靠性、RTSP 视频链路的改造计划。当前结论是：不把 Qt 上位机直接接入 ROS1 图，而是在下位机侧保留一个正式的 ROS1 bridge/gateway 节点；上位机继续使用边界协议访问下位机。

## 1. 目标架构

```text
Qt 上位机
  ├─ TCP/JSON：控制、状态、心跳、错误、camera_info
  └─ RTSP：多路视频流

ROS1 下位机
  ├─ host_bridge_node：JSON <-> ROS topic/service/action
  ├─ watchdog / ack / 状态同步
  └─ camera pipeline：RTSP 推流
```

核心原则：

- 上位机不直接依赖 ROS1 runtime、ROS master、环境变量和 ROS 网络发现。
- 下位机内部继续使用 ROS1 topic/service/action。
- `host_bridge_node` 作为 Qt 上位机和 ROS1 系统之间的唯一边界。
- 视频流继续走 RTSP，不走 TCP/JSON，也不优先走 ROS image topic。
- 未来如果迁移 ROS2，尽量只替换下位机 bridge 实现，上位机协议保持稳定。

## 2. 当前实现判断

当前 TCP/JSON 实现可以用于联调和半成品演示，但不应直接视为实机级可靠链路。

当前优点：

- 使用 Qt 异步 `QTcpSocket`，不会在连接时阻塞 UI。
- TCP 收包有缓冲区，按换行拆 JSON，能处理粘包和半包。
- 发送 JSON 统一加 `\n` 作为帧边界。
- 通信层通过 Qt signal 向 Controller/UI 转发数据。
- 基础心跳、重连、统计信息已经存在。

当前不足：

- 自动重连、手动断开语义还需要继续梳理。
- 心跳已经具备 `heartbeat` / `heartbeat_ack` 识别、RTT 统计和超时计数。
- 发送帧已经具备 `seq`，关键命令已经有 ACK pending/成功/失败/超时上抛；接收侧已有最大帧限制和基础 schema 校验，完整 schema 还可继续加深。
- 上位机连接/重连后会主动发送 `sync_request` 和 `camera_list_request`，并接收 `system_snapshot` / `camera_list_response`。
- RTSP 播放已切换为 FFmpeg 子进程低缓冲拉流，再由 Qt Widget 渲染；仍需继续补断流检测和自动重连。

## 3. 命令分类与 ACK 策略

不是所有上位机命令都应该逐条 ACK。命令按语义分为三类。

### 3.1 高频流式控制：不逐条 ACK

这类命令频率高，后一条自然覆盖前一条。逐条 ACK 会增加延迟、拥塞和实现复杂度。

典型命令：

- `operator_input`：操作者输入状态，统一承载键盘、手柄或后续其他输入设备的当前意图。
- 键盘按住产生的连续输入状态。
- 手柄摇杆连续输入状态。
- 机械臂模式下的连续末端输入意图。
- 高频关节微调输入意图，如果后续做成连续按压发送。

上位机不再直接下发 `/cmd_vel` 语义。上位机只下发操作者输入状态；下位机 `host_bridge_node` 根据当前模式、机器人状态、安全约束和输入映射，自行解析为 ROS `/cmd_vel`、机械臂控制 topic 或其他具体执行指令。

不要下发 Qt 原始按键码。协议应使用稳定的协议层输入名，避免下位机依赖 Qt key code、键盘布局或上位机 UI 实现。

`operator_input` 是一份完整输入快照，可以同时包含键盘和手柄：

- 键盘使用“当前按下集合”，未出现在集合中的键视为未按下。
- 手柄使用“完整状态快照”，按钮、摇杆、扳机、方向键等值都放入消息中。
- 下位机维护输入映射表，负责解释 `w/s/a/d`、手柄轴、按钮等输入代表的运动或动作。
- 上位机只负责采集和发送输入状态，不负责把输入翻译成前进、后退、转向等运动语义。

建议字段：

```json
{
  "type": "operator_input",
  "protocol_version": 1,
  "seq": 12345,
  "timestamp_ms": 123456789,
  "ttl_ms": 300,
  "mode": "vehicle",
  "keyboard": {
    "pressed_keys": ["w", "d", "shift"]
  },
  "gamepad": {
    "connected": true,
    "buttons": {
      "a": false,
      "b": false,
      "x": false,
      "y": false,
      "start": false,
      "back": false,
      "lb": false,
      "rb": true,
      "l3": false,
      "r3": false,
      "dpad_up": false,
      "dpad_down": false,
      "dpad_left": false,
      "dpad_right": false
    },
    "axes": {
      "left_x": 0.0,
      "left_y": 0.82,
      "right_x": -0.31,
      "right_y": 0.0,
      "lt": 0.0,
      "rt": 0.5
    }
  }
}
```

键盘输入名建议使用小写稳定名字，例如 `w`、`a`、`s`、`d`、`space`、`shift`、`ctrl`、`up`、`down`、`left`、`right`、`1`、`6`。数字键直接使用字符数字；兼容期下位机 bridge 可接受旧的 `num_1` 形式。手柄按钮使用常见命名，`l3` 表示左摇杆按下，`r3` 表示右摇杆按下。手柄轴范围统一为 `[-1.0, 1.0]`，扳机范围统一为 `[0.0, 1.0]`。如果手柄未连接，可以发送 `"connected": false`，并将按钮和轴置为默认值。

可靠性依靠：

- `seq`：下位机丢弃旧序号命令。
- `ttl_ms`：输入状态过期后不再执行。
- 下位机 watchdog：一段时间没收到有效 `operator_input` 就自动停车或进入安全态。
- 状态切换、断开、急停时下位机立即清空输入意图并输出安全指令。

### 3.2 关键离散命令：必须 ACK

这类命令低频，但必须知道下位机是否收到并执行。

典型命令：

- `emergency_stop`
- `clear_emergency`
- `motor_init`
- `motor_enable`
- `motor_disable`
- `set_control_mode`
- `home_arm`
- `start_task`
- `stop_task`
- `set_param`
- 离散型执行器动作，例如 `executor_action`

命令示例：

```json
{
  "type": "motor_init",
  "protocol_version": 1,
  "seq": 2001,
  "timestamp_ms": 123456789,
  "params": {}
}
```

ACK 示例：

```json
{
  "type": "ack",
  "protocol_version": 1,
  "ack_type": "motor_init",
  "seq": 2001,
  "ok": true,
  "code": 0,
  "message": "motor initialized",
  "timestamp_ms": 123456999
}
```

UI 侧需要显示这些状态：

- 等待确认。
- 成功。
- 失败，展示 `code` 和 `message`。
- 超时。

上位机内部 ACK 生命周期事件：

- `ack_pending`：关键命令已发出，正在等待下位机 ACK。
- `ack`：下位机返回确认，`ok=true` 表示成功，`ok=false` 表示失败。
- `ack_timeout`：超过 ACK 超时时间仍未收到匹配 ACK。
- `ack_disconnected`：等待 ACK 期间 TCP 断开。
- `ack_unmatched` / `ack_mismatch`：收到迟到 ACK、未知 seq ACK 或 ACK 类型与 pending 命令不一致。

### 3.3 查询/同步命令：必须 response

这类命令不是执行动作，而是请求状态或能力信息。它们应返回对应 response，不只返回普通 ACK。

典型命令：

- `sync_request`
- `camera_list_request`
- `system_snapshot_request`
- `get_param`
- `ping`
- `heartbeat`

示例：

```json
{
  "type": "camera_list_request",
  "protocol_version": 1,
  "seq": 3001,
  "timestamp_ms": 123456789
}
```

返回：

```json
{
  "type": "camera_list_response",
  "protocol_version": 1,
  "seq": 3001,
  "cameras": []
}
```

## 4. 协议 v1 帧清单

本节作为上位机和 `host_bridge_node` 的第一版对接规格。字段命名以这里为准：统一使用 `protocol_version`、`seq`、`timestamp_ms`，不再新增旧式 `timestamp` 字段。

### 4.1 高频输入快照：operator_input

高频输入快照不逐条 ACK。下位机按 `seq` 丢弃旧帧，按 `ttl_ms` 判断输入是否过期。

```jsonc
{
  "type": "operator_input",        // 固定值：操作者输入快照
  "protocol_version": 1,           // 协议版本
  "seq": 12345,                    // 上位机单调递增序号
  "timestamp_ms": 123456789,       // 上位机发送时的毫秒时间戳
  "ttl_ms": 300,                   // 输入有效期，超时后下位机必须进入安全输出
  "mode": "vehicle",               // 当前上位机控制模式：vehicle / arm
  "keyboard": {
    "pressed_keys": ["w", "d", "shift"] // 当前按下集合；未出现的键视为未按下
  },
  "gamepad": {
    "connected": true,             // 手柄是否在线
    "buttons": {
      "a": false,                  // A 键
      "b": false,                  // B 键
      "x": false,                  // X 键
      "y": false,                  // Y 键
      "start": false,              // Start 键
      "back": false,               // Back 键
      "lb": false,                 // 左肩键
      "rb": true,                  // 右肩键
      "l3": false,                 // 左摇杆按下
      "r3": false,                 // 右摇杆按下
      "dpad_up": false,            // 方向键上
      "dpad_down": false,          // 方向键下
      "dpad_left": false,          // 方向键左
      "dpad_right": false          // 方向键右
    },
    "axes": {
      "left_x": 0.0,               // 左摇杆 X，范围 [-1.0, 1.0]
      "left_y": 0.82,              // 左摇杆 Y，范围 [-1.0, 1.0]
      "right_x": -0.31,            // 右摇杆 X，范围 [-1.0, 1.0]
      "right_y": 0.0,              // 右摇杆 Y，范围 [-1.0, 1.0]
      "lt": 0.0,                   // 左扳机，范围 [0.0, 1.0]
      "rt": 0.5                    // 右扳机，范围 [0.0, 1.0]
    }
  }
}
```

说明：

- 键盘和手柄同时存在，二者都在同一帧里下发。
- 键盘只发按下集合，不发完整布尔表。
- 手柄发完整快照，因为下位机需要同时解析按钮、摇杆、扳机和方向键。
- `w/s/a/d`、`l3/r3` 等名字是协议层稳定名字，不是 Qt key code。
- 急停按键仍应额外触发 `emergency_stop` 关键命令，不能只依赖输入快照。

### 4.11 协议错误：protocol_error

下位机或 mock server 收到不支持的消息类型时，不应静默兼容旧协议，也不应直接断开 TCP。建议返回 `protocol_error`，方便上位机暴露问题。

```jsonc
{
  "type": "protocol_error",
  "protocol_version": 1,
  "seq": 4001,
  "code": 1001,
  "message": "unsupported message type: cmd_vel",
  "timestamp_ms": 123456999
}
```

## 5. 急停可靠性原则

急停不能只依赖某一次 TCP JSON 命令。

建议采用：

```text
急停可靠性 = 命令 ACK + 状态广播 + 下位机 watchdog + 硬件急停链路
```

具体要求：

- 上位机发送 `emergency_stop`，必须等待 ACK。
- 下位机一旦进入急停状态，持续广播 `emergency_state`。
- 如果 ACK 丢失，但收到 `emergency_state`，UI 仍应显示下位机已急停。
- 如果 ACK 和状态广播都没有收到，UI 显示“急停确认超时”，但上位机本地仍进入安全态。
- 下位机侧急停处理优先级必须高于普通控制命令。

## 6. 通信改造阶段

### 阶段 1：TCP/JSON 基础可靠性

目标：让现有链路从“能联调”变成“可长期联调”。

- 修复自动重连计数。（已完成）
- 区分主动断开和异常断开，主动断开不自动重连。（已完成）
- 心跳改成 `heartbeat` / `heartbeat_ack`。（已完成）
- 心跳携带 `seq`、`timestamp`。（已完成）
- 增加协议版本字段，例如 `protocol_version: 1`。（已完成发送侧）
- 增加最大帧长度限制，防止异常 JSON 撑爆内存。（已完成）
- 对关键接收消息做字段校验和范围校验。（已完成基础版本）

### 阶段 2：关键命令 ACK

目标：让低频关键动作可确认、可超时、可诊断。

- 为关键离散命令增加 `seq`。（已完成基础发送侧）
- 增加 ACK 管理器，维护 pending command。（已完成基础版本）
- UI 显示 ACK 等待、成功、失败、超时。（已完成日志层展示）
- 失败 ACK 带 `code`、`message`。（已完成基础解析）
- `operator_input` 不逐条 ACK，只加 `seq`、`ttl_ms`。

### 阶段 3：状态同步与下位机 bridge 规范

目标：重连后状态一致，下位机 ROS1 细节隐藏在 bridge 内。

- 定义正式 `host_bridge_node` 职责。（文档已定义，代码按该边界实现）
- 下位机启动后发送 `hello` / `capabilities`。（上位机已接收并展示，mock 已支持）
- 上位机连接或重连后发送 `sync_request`。（已完成）
- 下位机返回 `system_snapshot`。（上位机已接收并展示摘要，mock 已支持）
- `system_snapshot` 至少包含：
  - 当前控制模式。
  - 急停状态。
  - 电机初始化/使能状态。
  - 各模块在线状态。
  - 摄像头列表。
  - 最近错误码。
- 下位机重连后重新发送所有 `camera_info` 或响应 `camera_list_request`。（上位机已主动请求并合流到摄像头信息）

### 阶段 4：下位机 watchdog 与安全态

目标：上位机异常时，下位机自动进入安全状态。

- 下位机超过指定时间未收到有效 `operator_input`，自动停车或进入安全态。
- 下位机丢弃过期 `ttl_ms` 输入状态。
- 下位机丢弃旧 `seq` 输入状态。
- 连接断开时，下位机进入安全态。
- 模式切换、初始化、使能等关键动作必须经过确认。

## 7. RTSP 视频链路改造阶段

视频流继续走 RTSP。TCP/JSON 只负责传 `camera_info`、摄像头列表、视频状态和错误信息。

详细方案见 [视频推流与摄像头管理方案.md](视频推流与摄像头管理方案.md)。当前落地顺序是先做无处理摄像头的硬件直推 RTSP，再接入两路 OpenCV C++ 后处理画面。

### 阶段 0：下位机无处理直推

- 新增 `video_manager_node`，负责读取摄像头配置、启停 FFmpeg/GStreamer 推流进程、监控进程健康。
- 无处理摄像头直接从 `/dev/videoX`、CSI 或网络摄像头进入 RTSP 管线，不经过 ROS image topic。
- `host_bridge_node` 从 video manager 获取 camera list，响应 `camera_list_request`，并在状态变化时主动推 `camera_info`。
- 第一版可用 MediaMTX 作为 RTSP server，推流工具先用 FFmpeg 或 GStreamer，默认使用 RTSP/TCP 发布，后续按硬件能力调整为硬编码低延迟管线。

### 阶段 1：播放器状态机

为每路 `RtspPlayerWidget` 增加状态：

```text
Offline
Connecting
Playing
Stalled
Error
Reconnecting
```

需要监听：

- FFmpeg 子进程启动、退出和 stderr。
- raw frame 到达时间。
- 播放开始/停止状态。

### 阶段 2：断流检测和自动重连

- 播放失败后自动重试。
- 重试使用退避策略，例如 1s、2s、5s。
- 启动后 N 秒仍未进入播放态，判定为连接超时。
- 每路显示最后错误、重连次数、当前状态。

### 阶段 3：摄像头列表同步

- 增加 `camera_list_request`。
- 增加 `camera_list_response`。
- 下位机摄像头上线/离线时主动推 `camera_info`。
- 上位机重连后主动请求 camera list。

### 阶段 4：低延迟方案落地

当前已使用 FFmpeg 子进程低缓冲拉流，避免 `QMediaPlayer` 的默认播放缓冲。构建不链接 `libav*`，运行时只需要 `ffmpeg` 可执行文件。

后续可继续优化：

- 增加 FFmpeg 退出后的自动重连和退避策略。
- 增加帧到达频率、最近错误和解码进程状态显示。
- 如果 CPU 压力过高，再评估硬解、GStreamer 或平台专用渲染。

## 11. 落地 commit 计划

下面是本轮开始落地的顺序。每个 commit 都应该能独立构建，避免一次性混杂太多行为变化。

1. `docs(protocol): define host bridge protocol v1`
   - 补齐 `operator_input`、关键命令 ACK、心跳、同步、摄像头列表和主动状态帧。
   - 明确键盘为按下集合，手柄为完整快照，手柄按键使用 `l3/r3`。

2. `refactor(protocol): introduce host message builders`
   - 新增 `src/communication/HostProtocol.h/.cpp`。
   - 集中生成 `protocol_version`、`seq`、`timestamp_ms` 和基础 JSON 帧。
   - 保留旧接收字段兼容，但新增发送帧按协议 v1 生成。

3. `feat(input): add operator input snapshot model`
   - 新增协议层 `OperatorInputState` 数据结构。
   - 键盘输出 `pressed_keys` 集合。
   - 手柄输出完整 buttons/axes 快照。

4. `feat(communication): send operator input instead of cmd_vel`
   - 上位机高频输入改发 `operator_input`。
   - 停止由上位机把键盘/手柄翻译成 `/cmd_vel`。
   - 键盘和手柄允许同时存在，同一帧同时下发。

5. `feat(protocol): add basic heartbeat ack handling`
   - 心跳改为协议 v1 帧。
   - 接收 `heartbeat_ack` 后更新心跳在线状态。
   - mock server 支持 `heartbeat_ack`。

6. `test(mock): support protocol v1 operator input`
   - mock server 识别 `operator_input`。
   - 对关键命令返回基础 ACK。
   - smoke 脚本继续覆盖构建与无阻塞检查。

7. `feat(protocol): track critical command ack lifecycle`
   - 发送关键命令时登记 pending ACK。
   - 收到 ACK 时匹配 seq 并上抛成功/失败。
   - 超时、断线、迟到或不匹配 ACK 都生成明确事件。
   - mock server 支持 `MOCK_ACK_MODE=drop|fail` 和 `MOCK_ACK_DELAY_SEC`。

8. `test(protocol): add protocol smoke coverage`
   - 增加 `scripts/protocol_smoke_test.py`。
   - 覆盖 `hello`、`capabilities`、`heartbeat_ack`、`operator_input`、关键命令 ACK、`sync_request`、`camera_list_request` 和 `protocol_error`。
   - `scripts/run_all_tests.sh` 默认运行 protocol smoke。

9. `harden(protocol): validate tcp frames`
   - 限制单帧最大长度。
   - 对 ACK、heartbeat、摄像头、状态、IMU、CO2 等关键接收帧做基础 schema 校验。
   - 增加协议错误计数、心跳 RTT 和心跳超时统计。
   - 主动断开不再触发自动重连。

10. `feat(protocol): sync bridge state on connect`
   - 连接/重连后自动发送 `sync_request` 和 `camera_list_request`。
   - 接收并展示 `hello`、`capabilities`、`system_snapshot`。
   - `camera_list_response` 合流到每路 `camera_info` 更新。

后续单独阶段再做：

- 更完整的通信诊断 UI 和 protocol smoke 坏包/半包/超大包自动化覆盖。
- RTSP 播放器状态机、断流重连和视频诊断面板。
